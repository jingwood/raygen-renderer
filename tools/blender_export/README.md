# blend → raygen scene converter

Convert a `.blend` file into a baseline raygen scene JSON plus a companion
`.obj` / `.mtl` pair.

## Usage

```bash
blender -b path/to/scene.blend -P blend_to_raygen.py -- \
    --out path/to/raygen_export/scene.json
```

Anything before `--` is consumed by Blender; everything after is forwarded to
the script. Output basename drives the `.obj`/`.mtl` filenames written next
to the JSON.

### Example (F2)

```bash
blender -b "F:/3D Models/F2/F2.blend" -P blend_to_raygen.py -- \
    --out "F:/3D Models/F2/raygen_export/F2.json"
```

## What gets exported

- **Geometry** — every visible `MESH` object is bundled into a single OBJ
  (`<basename>.obj`) with `forward = -Z, up = Y` axis conversion. Transforms
  are baked into the vertex stream, so the JSON `world.<basename>` entry sits
  at the origin with identity rotation/scale.
- **Camera** — `scene.camera` becomes `mainCamera` (location, Euler angle in
  degrees, vertical FoV, exposure, optional DoF).
- **World envmap** — Background → Environment Texture (with optional Mapping
  rotation) becomes the `envmap` block. Texture path is written relative to
  the output directory when sensible.
- **Materials** — Principled BSDF nodes are mapped to raygen material fields:
  - Base Color → `color` (and `tex` when a TEX_IMAGE feeds the socket)
  - Metallic → `metallic`
  - Roughness → `roughness`
  - Specular IOR Level → `glossy` (Blender 4.x naming; falls back to
    pre-4.x `Specular`)
  - Transmission → `refraction`, with IOR → `refractionRatio`
  - Alpha < 1 → `transparency`
  - Emission Color × Strength → `color` + `emission`
- **Lights** — `POINT` → tiny emissive sphere; `AREA` → emissive plane sized
  to the Blender shape; `SPOT` → emissive sphere with `spotRange`; `SUN` is
  skipped (use the envmap rotation/intensity instead).

## Caveats

- Light `energy` is in Watts and copies straight through to `emission`. It
  rarely matches the unit-less raygen scale 1:1 — expect to tune.
- Material → JSON mapping is a baseline. Glossy/roughness values often need
  manual nudges (see [F2.json](../../../../3D Models/F2/raygen_export/F2.json)
  for a reference of what hand-tuned output looks like).
- Volumetric / participating-media settings (`engineFlame`, vapor cones,
  etc.) are **not** generated by `blend_to_raygen.py`. Add those by hand
  after the converter runs, or use `vapor_paths_merge.py` for wing-vapor
  streamlines authored as Blender Empties (see below).
- `blend_to_raygen.py` overwrites the whole JSON — re-running the full
  exporter discards hand-edits. Use the marker-based merge tool below for
  incremental updates that preserve the rest of the scene.

## Wing-vapor path merge — `vapor_paths_merge.py`

For high-G wing vortex condensation (and other curved emissive volumes)
authored as Blender Empties.

### Authoring in Blender

Add Empty objects (any display type) named:

```
vapor.<groupName>.<index>     e.g. vapor.wingR.001, vapor.wingR.002, ...
```

- **Parent** every Empty to the owning mesh (typically the F2 mesh) so its
  local-space position matches what raygen sees with `pathFollowObject`.
- **Order** is taken from the trailing digit suffix — the streamline goes
  1 → 2 → ... → N. Any width works (`001` through `0001`); Blender's
  Ctrl+D auto-suffix `.001`, `.002`, ... drops in directly.
- **Tube radius** at each control point comes from the Empty's Custom
  Property `radius`; fallback is the Empty's `empty_display_size`.

Empties belonging to the same `<groupName>` form one streamline. Multiple
groups (`wingR`, `wingL`, `canardR`, ...) become separate emissive volumes.

### Marker block in the target JSON

Insert this pair once, inside the owning SceneObject in your scene JSON
(typically the `f2` block), at the spot where wing vapor entries should
appear:

```jsonc
"f2": {
    ...,
    // <BLENDER_VAPOR_BEGIN>
    // <BLENDER_VAPOR_END>
},
```

Everything outside the markers is preserved byte-for-byte.

### Running

```bash
blender -b "F:/3D Models/F2/F2.blend" -P vapor_paths_merge.py -- \
    --target "F:/3D Models/F2/raygen_export/F2.json"
```

Add `--dry-run` to print the would-be replacement to stdout without
touching the file. Re-running re-emits the marker region from the current
.blend state — hand-edits *inside* the markers will be lost; tune defaults
in the script's `TUNING` dict (top of file) for persistent overrides.
